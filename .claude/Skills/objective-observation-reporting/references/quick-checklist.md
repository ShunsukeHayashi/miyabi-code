# Objective Observation Quick Reference

## 🚦 即座チェック（レポート送信前に確認）

### ❌ 禁止表現（使用したら即修正）

| 禁止 | 代替表現 |
|------|---------|
| 致命的な欠陥 | 影響のある現象 |
| 設計ミス | 観察された動作 |
| 明らかなバグ | 想定と異なる動作 |
| 〜に違いない | 〜の可能性がある |
| 〜するべきだ | 〜を検討いただければ幸いです |
| 最後通告 | フィードバック / 観察レポート |
| 緊急性が高い | 影響範囲として〜 |

### ✅ 推奨表現（積極的に使用）

```
「観察された現象として」
「〜という傾向が見られた」
「〜の可能性がある」
「内部設計は不明だが」
「検討いただければ幸いです」
「意図された動作かどうか不明」
「再現条件は未検証」
「詳細は非公開のため推測を控える」
```

## 📋 5秒チェックリスト

レポート作成後、以下を5秒で確認：

1. [ ] **断定** → 「〜である」を「〜と観察された」に
2. [ ] **感情** → 「致命的」「緊急」などを削除
3. [ ] **命令** → 「すべき」を「検討いただければ」に
4. [ ] **推測** → 「おそらく」「〜だろう」と明記
5. [ ] **不明** → 知らないことは「不明」と書く

## 📝 最小テンプレート

```markdown
## 観察レポート: [タイトル]

### 現象
[客観的に何が起きたか]

### 影響
[ユーザー体験への具体的影響]

### 不明点
- 内部設計の詳細は非公開
- 意図された動作かどうか不明

### 要望（あれば）
[検討いただければ幸いです]
```

## 🔄 変換例

```
Before: 「コンテキスト圧縮のrecency biasは設計欠陥だ」
After:  「コンテキスト圧縮後、直近の内容がサマリーの中心になる傾向を観察した。
         内部設計の詳細は不明。」
```

## 🎯 適用場面

このスキルを適用すべき場面：

1. **システム動作の報告** - 想定と異なる動作を報告する場合
2. **フィードバック作成** - 開発者やサポートへの報告
3. **トラブルシューティング** - 問題の記録と共有
4. **ユーザーへの説明** - 技術的制限の説明

## ⚠️ 特に注意が必要な場面

1. **長時間作業後** - 疲労で感情的になりやすい
2. **フラストレーション時** - 断定的表現が増えやすい
3. **急いでいる時** - チェックを省略しがち

→ **必ず5秒チェックリストを実行**
