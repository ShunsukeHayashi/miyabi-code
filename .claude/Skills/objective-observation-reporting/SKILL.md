# Objective Observation Reporting

事実と推測を明確に区別し、客観的なフィードバック・観察レポートを作成するためのプロトコル。

## 背景：このスキルが必要な理由

### 問題事例（避けるべきパターン）

```markdown
❌ 悪い例：
「コンテキスト圧縮アルゴリズムにrecency biasがあり、
primary objectiveが保護されない致命的な設計欠陥がある」

問題点：
1. 「recency bias」→ 内部設計を知らないのに断定
2. 「致命的な設計欠陥」→ 感情的・誇張的な表現
3. アルゴリズムの詳細を推測で語っている
```

### 正しいアプローチ

```markdown
✅ 良い例：
「長時間セッションでコンテキスト圧縮が発生した後、
セッション開始時の主目的よりも直近の作業内容が
サマリーに反映される傾向を観察しました。
これが意図された動作かどうかは不明です。」

改善点：
1. 観察した事実のみを記述
2. 内部設計への断定を避ける
3. 感情的な表現を排除
```

## P0: 絶対原則

### P0.1 事実と推測の明確な区別

| 区分 | 使用可能な表現 | 禁止表現 |
|------|--------------|---------|
| **事実** | 「観察した」「確認した」「発生した」 | 推測を含む断定 |
| **推測** | 「可能性がある」「考えられる」「不明」 | 「〜である」「〜に違いない」 |
| **不明** | 「内部設計は非公開」「詳細は不明」 | 「〜の設計欠陥」 |

### P0.2 禁止表現リスト

以下の表現は**絶対に使用禁止**：

```
❌ 「致命的な欠陥」
❌ 「設計ミス」
❌ 「明らかなバグ」
❌ 「〜に違いない」
❌ 「〜するべきだ」（命令的な要求）
❌ 「最後通告」
❌ 「緊急性が高い」（根拠なく）
```

### P0.3 推奨表現リスト

以下の表現を**積極的に使用**：

```
✅ 「観察された現象として」
✅ 「〜という傾向が見られた」
✅ 「〜の可能性がある」
✅ 「内部設計は不明だが」
✅ 「検討いただければ幸いです」
✅ 「意図された動作かどうか不明」
✅ 「再現条件は未検証」
```

## P1: 事象の記述プロトコル

### P1.1 具体→抽象の3段階変換

```
【レベル1: 具体的事象】
「Dioxus Labs戦略検討からYouTube Live準備へ進行中に、
コンテキスト圧縮後、開始時の目的より途中の派生作業が
サマリーの中心になっていた」

↓ 一般化

【レベル2: 一般化した記述】
「長時間セッションでコンテキスト圧縮が発生した際に、
直近の作業内容がサマリーの中心となり、
セッション開始時の主目的への参照が薄くなる場合がある」

↓ 抽象化

【レベル3: 抽象化した概念】
「コンテキスト管理における『時間的近接性』と
『目的的重要性』の優先度バランスに関する課題」
```

### P1.2 記述テンプレート

```markdown
## 観察レポート

### 観察された現象
[具体的に何が起きたかを客観的に記述]

### 影響
[ユーザー体験やワークフローへの具体的な影響]

### 再現条件（わかる範囲で）
- セッション時間: [例: 2-3時間]
- 作業内容: [例: 複数の派生タスクを含む]
- 再現頻度: [例: 未検証 / 1回のみ / 複数回]

### 不明な点
- [内部設計に関する不明点を明記]
- [意図された動作かどうか]

### 要望（あれば）
[「検討いただければ幸いです」形式で記述]

### 備考
[推測や感想があれば、明確に「推測」とラベル付けして記述]
```

## P2: 自己チェックリスト

レポート作成前に以下を確認：

```
□ 事実と推測を明確に区別しているか
□ 内部設計について断定していないか
□ 感情的・誇張的な表現を使っていないか
□ 「不明」なことを「不明」と明記しているか
□ 命令的ではなく要望形式になっているか
□ 具体事象を適切に一般化しているか
□ 禁止表現リストに該当する表現がないか
```

## P3: 例外ケース

### 例外1: 明確なドキュメント違反の場合

公式ドキュメントに明記された仕様と異なる動作の場合のみ、
「仕様と異なる動作」と記述可能：

```markdown
✅ 「公式ドキュメント[URL]に記載された仕様では〜となっているが、
   実際の動作は〜となっている」
```

### 例外2: 再現可能なエラーの場合

エラーメッセージやスタックトレースがある場合は引用可能：

```markdown
✅ 「以下のエラーが発生した：
   [エラーメッセージをそのまま引用]」
```

## 実践例

### Before（修正前）

```markdown
## 致命的なバグレポート：コンテキスト圧縮の設計欠陥

コンテキスト圧縮アルゴリズムにはrecency biasがあり、
primary objectiveを保護しない致命的な設計欠陥がある。
これは明らかに修正すべきバグである。
```

### After（修正後）

```markdown
## 観察レポート：長時間セッションでのコンテキスト管理

### 観察された現象
長時間（2-3時間）の作業セッションでコンテキスト圧縮が
発生した後、セッション開始時の主目的よりも直近の作業内容が
サマリーに反映される傾向を経験しました。

### 影響
当初の目的に立ち返る際に追加の説明が必要になりました。

### 不明な点
- コンテキスト圧縮の内部ロジックは非公開
- これが意図された動作かどうかは不明
- 再現条件や頻度は未検証

### 要望（検討いただければ幸いです）
セッション開始時の目的を長期的に参照しやすくする仕組み、
またはユーザーが重要なコンテキストを明示的にマークできる機能

### 備考
内部設計に関する判断は控えさせていただきます。
```
