export default async function hello() {

miyabi run hello

miyabi publish

## 🚨 6. クライシスマネジメント

### 6.1 炎上対策プロトコル

#### 6.1.1 炎上レベル分類

```yaml
crisis_levels:
  Level 1 (軽微):
    定義: 個別のネガティブコメント (<10件)
    例: 機能に対する不満、使い方がわからない
    対応: コミュニティマネージャーが個別対応
    対応時間: 1時間以内

  Level 2 (中程度):
    定義: 複数のネガティブ投稿 (10-50件)、メディア言及なし
    例: バグによる不具合、ドキュメント不足
    対応: チームミーティング → 公式声明
    対応時間: 3時間以内

  Level 3 (重大):
    定義: 広範な批判 (50+件)、メディア言及、トレンド入り
    例: セキュリティ脆弱性、データ漏洩、炎上発言
    対応: 経営陣関与 → 謝罪声明 → 記者会見 (必要に応じて)
    対応時間: 1時間以内
```

---

#### 6.1.2 炎上対応フロー (Level 2-3)

```yaml
crisis_response_flow:
  Step 1: 即時対応 (0-30分):
    - SNS投稿一時停止
    - 状況把握 (原因・規模・影響)
    - 緊急ミーティング招集

  Step 2: 初期声明 (30分-1時間):
    - 事実確認中であることを表明
    - 謝罪 (必要に応じて)
    - 対応予定時刻を明示

    例:
    "We're aware of [問題] and investigating.
     We'll provide an update by [時刻].
     We sincerely apologize for any inconvenience."

  Step 3: 原因調査・対策立案 (1-3時間):
    - 技術チームが原因特定
    - 対策方法決定
    - 声明文ドラフト作成

  Step 4: 公式声明 (3-6時間):
    - ブログ記事公開
    - X, LinkedIn, Discord で告知
    - メディア対応 (必要に応じて)

    声明内容:
    1. 謝罪
    2. 原因説明
    3. 対策内容
    4. 再発防止策
    5. 補償 (必要に応じて)

  Step 5: 継続対応 (6時間-数日):
    - 個別問い合わせ対応
    - 進捗アップデート (6時間ごと)
    - 完全復旧まで監視

  Step 6: 事後分析 (1週間後):
    - ポストモーテム実施
    - 再発防止策実装
    - チーム内共有
```

### 6.2 ネガティブフィードバック対応

#### 6.2.1 ネガティブコメント分類

```yaml
negative_feedback_types:
  Type 1_建設的批判:
    特徴: 具体的な改善提案、冷静
    例: 「機能Aが使いにくい。こう改善してほしい」
    対応:
      - 感謝を示す
      - 真摯に受け止める
      - 改善検討を約束
      - GitHub Issue作成を提案

    返信例:
    "Thank you for the feedback!
     We appreciate your suggestion on [機能].
     We'll discuss this with the team.
     Feel free to open an Issue: [URL]"

  Type 2_不満・愚痴:
    特徴: 感情的、具体性欠ける
    例: 「使いにくい」「高すぎる」
    対応:
      - 共感を示す
      - 具体的な問題をヒアリング
      - DM誘導 (必要に応じて)

    返信例:
    "We're sorry to hear that.
     Could you share more about what's frustrating?
     We'd love to help improve your experience.
     DM us anytime."

  Type 3_誤解:
    特徴: 事実誤認に基づく批判
    例: 「機能Xがない」(実際はある)
    対応:
      - 丁寧に訂正
      - ドキュメントURL提示
      - 改善余地があれば認める

    返信例:
    "Actually, we do support [機能X]!
     Here's the doc: [URL]
     We'll improve our onboarding to make this clearer.
     Thanks for pointing it out!"

  Type 4_荒らし・アンチ:
    特徴: 攻撃的、非建設的、繰り返し
    例: 「クソツール」「詐欺」
    対応:
      - 1回目: 冷静に返信
      - 2回目: 無視
      - 3回目: ブロック or ミュート

    返信例 (1回目):
    "We're sorry you feel that way.
     If you have specific concerns, we're happy to address them."

    (2回目以降: 無視)
```
